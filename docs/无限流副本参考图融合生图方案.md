### 1\.3 目标画风总结



综合 5 张参考图，目标画风可以提炼为一个**可插拔的风格描述系统**：



```Plain Text
核心画风定位: 东方奇幻 × 日系美学 × 电影叙事
= Eastern Fantasy Anime Cinematic (EFAC)

不是一个固定的"风格"，而是一个"风格参数空间"：
- 写实度: 可从日系平涂 → 厚涂概念艺术之间滑动

- 情绪: 孤独美学，一个人面对广阔世界的渺小与壮丽
```

## 2\. 画风提炼与风格词汇表



### 2\.1 结构化风格描述体系



将画风拆解为 **7 个可独立调控的维度**，每个维度提供一个从参考图中提炼的关键词词库：



```TypeScript
interface ArtStyleProfile {
  // 1. 渲染风格 — 控制整体画风倾向
  render_style: "anime_cel" | "semi_realistic" | "concept_art" | "oil_painting";

  // 2. 线条处理 — 控制边缘和轮廓
  line_work: "sharp_cel_outline" | "soft_blend" | "painterly_no_line" | "textured_sketch";

  // 3. 光影风格 — 控制明暗关系
  lighting: "soft_diffused" | "dramatic_chiaroscuro" | "volumetric_godray" | "rim_backlight";

  // 4. 色彩策略 — 控制色调和饱和度
  color_strategy: "cool_pastel" | "warm_saturated" | "cold_warm_contrast" | "dark_accent_glow";

  // 5. 背景处理 — 控制环境复杂度
  background: "minimal_gradient" | "blurred_env" | "detailed_fantasy" | "cinematic_landscape";

  // 6. 特效元素 — 标志性视觉元素
  effects: ("particle_dust" | "fog_mist" | "godray" | "glowing_creature" | "petals" | "aurora")[];

  // 7. 情绪基调 — 控制整体氛围
  mood: "serene_gentle" | "mysterious_lonely" | "epic_adventurous" | "warm_youthful";
}
```

## 3\. 现有管线诊断与问题分析



### 3\.1 当前 Prompt 构建流程



```Plain Text
当前流程 (adventure-image.ts):

scene_visual_brief (LLM 生成)
  ├── subject: string (max 80 chars)
  ├── environment: string (max 120 chars)
  ├── mood: enum (5种)
  ├── palette: enum (3种)
  └── composition: enum (3种)
          │
          ▼
  buildChapterPrompt()
  ├── 硬编码的构图/光影/饱和度模板
  ├── avatarVisualProfile.image_prompt_fragment (通用文字描述)
  ├── pathMemory.recurring_symbols
  └── commonNegative
          │
          ▼
  generateImage(prompt, config)  →  只发文字，无参考图
```



### 3\.2 诊断出的 6 个核心问题



|\#|问题|影响|严重度|
|---|---|---|---|
|1|**无画风锚定**|每次生成的画风随机，无法保证一致性|🔴 严重|
|2|**无参考图输入**|用户上传的人设图仅提取文字描述，实际图片从未被送入生图模型|🔴 严重|
|3|**Prompt 模板过于通用**|"cinematic illustration, contemporary Chinese urban life" 无法引导出特定画风|🟡 中等|
|4|**512 字符截断无策略**|StepFun 限制下直接截断，可能截断关键描述|🟡 中等|
|5|**palette 只有 3 种**|cyan\_blue/green\_cyan/pink\_purple 无法覆盖暗系/奇幻/史诗场景|🟡 中等|
|6|**无限流副本无独立视觉逻辑**|无限流的世界观更宏大、场景更多变，但共用同一套 prompt 模板|🟡 中等|



### 3\.3 512 字符预算分析



StepFun 的 512 字符限制是一个硬约束。当前 prompt 平均长度约 400\-600 字符，经常被截断。需要重新设计一个 **分层优先级** 的 prompt 构建策略：



```Plain Text
512 字符预算分配 (优先级从高到低):

[必选] 风格锚定句        ~200 chars  (画风一致性)
[必选] 场景主体+环境      ~120 chars  (剧情核心)
[可选] 光影/构图/色调      ~80 chars   (视觉品质)
[可选] 情绪/氛围          ~40 chars   (情感共鸣)
[可选] 特效元素           ~40 chars   (视觉丰富度)
[可选] 角色描述           ~60 chars   (仅当有空间时)
─────────────────────────────────────
总计上限                 512 chars
```

### 4\.2 模块设计



#### Module 1: `lib/style-extractor.ts` — 风格提取器



**输入**: `image/` 目录下的参考图

**输出**: `ArtStyleProfile` 结构化数据



**策略**: 使用 LLM \(gemini/gpt\-4v\) 对参考图做视觉分析，输出结构化的风格描述。这是一次性离线操作，结果缓存到 JSON 文件。



```TypeScript
// 定义
interface ArtStyleProfile {
  id: string;
  name: string;                           // "EFAC-v1" (Eastern Fantasy Anime Cinematic)
  anchor_prompt: string;                  // 通用风格锚定句 (≤200 chars)
  render_style: string;                   // "semi-realistic anime cinematic"
  lighting_signature: string;             // "volumetric godray + rim backlight"
  color_palette: string;                  // "cold-warm contrast with dark base"
  signature_effects: string[];            // ["fog_mist", "particle_dust", "godray"]
  negative_additions: string;             // 风格特定的负面提示
  chapter_style_modifiers: [              // 5 章各自的风格微调
    string, string, string, string, string
  ];
  reference_images: string[];             // 参考图路径列表
}
```



#### Module 2: `lib/prompt-budget.ts` — Prompt 预算分配器



**职责**: 在 512 字符硬限制下，智能分配 prompt 各部分的空间。



```TypeScript
interface PromptBudget {
  total: number;          // 512
  style_anchor: number;   // ~180 (固定)
  scene_content: number;  // ~160 (优先级最高)
  lighting: number;       // ~60
  mood: number;           // ~40
  effects: number;        // ~40
  safety_margin: number;  // ~32 (防止截断关键内容)
}
```



**核心算法**:

1. 固定分配 style\_anchor \(不可截断\)

2. 按优先级填充 scene\_content → lighting → mood → effects

3. 如果超预算，从 effects 开始逐级裁剪

4. 最终检查长度，确保 ≤512

    

#### Module 3: `lib/adventure-image.ts` — 改造现有 Prompt 构建器



**改造点**:

- `buildHeroPrompt()` / `buildChapterPrompt()` / `buildEndingPrompt()` 接受 `ArtStyleProfile` 参数

- 将硬编码的 "cinematic illustration, contemporary Chinese urban life" 替换为风格锚定句

- 使用 `PromptBudget` 做长度控制

- 为无限流副本增加独立的视觉逻辑

    

#### Module 4: `lib/image-api.ts` — 扩展支持参考图输入



**改造点**:

- `generateImage()` 新增可选的 `referenceImage` 参数

- 当 StepFun 支持参考图时，将参考图 base64 一并发送

- 当 API 不支持参考图时，仅使用风格化的 text prompt

    

### 4\.3 无限流副本专属视觉逻辑



无限流副本的世界观不同于标准副本 \(ability/growth/imagination\)。需要根据副本主题动态调整视觉风格：



```TypeScript
// 无限流副本的视觉风格映射
const INFINITE_ADVENTURE_STYLE_MAP: Record<string, Partial<ArtStyleProfile>> = {
  // 暗黑/悬疑主题 → 暗系奇幻风格 (参考 role2.png)
  dark_mystery: {
    render_style: "concept_art",
    lighting_signature: "dramatic chiaroscuro + bioluminescent glow",
    color_palette: "dark_accent_glow, deep blue-green with warm orange accents",
    signature_effects: ["fog_mist", "glowing_creature", "particle_dust"],
  },
  // 都市/现实主题 → 半写实风格 (参考 001.png)
  urban_realistic: {
    render_style: "semi_realistic",
    lighting_signature: "soft diffused natural light",
    color_palette: "warm_saturated, amber and neutral tones",
```

### 5\.2 Phase 2: Prompt 预算系统 \(1 天\)



**目标**: 在 512 字符限制下智能分配空间。



```TypeScript
// lib/prompt-budget.ts

interface PromptSegment {
  content: string;
  priority: number;   // 1 = 最高, 5 = 最低
  label: string;
}

/**
 * 在 maxLen 字符限制下，按优先级拼接 prompt 片段。
 * 高优先级片段保证完整保留，低优先级可能被裁剪。
 */
export function buildBudgetedPrompt(
  segments: PromptSegment[],
  maxLen: number = 512,
): string {
  // 按优先级排序 (priority 越小越优先)
  const sorted = [...segments].sort((a, b) => a.priority - b.priority);
  const result: string[] = [];
  let remaining = maxLen;

  for (const seg of sorted) {
    const sep = result.length > 0 ? ", " : "";
    const cost = seg.content.length + sep.length;
    if (cost <= remaining) {
      result.push(seg.content);
      remaining -= cost;
    } else if (remaining > 50) {
      // 尝试截断当前片段以适应剩余空间
      const truncated = seg.content.slice(0, remaining - sep.length - 3) + "...";
      result.push(truncated);
      break;
    } else {
      break;
    }
  }

  return result.join(", ");
}
```

```TypeScript
// lib/image-api.ts (扩展)

interface GenerateImageOptions {
  prompt: string;
  config: ImageApiConfig;
  /** 参考图 base64 数据 (用于风格参考) */
  referenceImage?: {
    b64Data: string;
    mimeType: string;
    /** 参考图对生成结果的影响强度 0-1 */
    strength?: number;
  };
}

export async function generateImage(
  promptOrOptions: string | GenerateImageOptions,
  maybeConfig?: ImageApiConfig,
): Promise<GenerateImageResult> {
  // 兼容旧调用方式
  const opts: GenerateImageOptions = typeof promptOrOptions === "string"
    ? { prompt: promptOrOptions, config: maybeConfig! }
    : promptOrOptions;

  const { prompt, config, referenceImage } = opts;

  // 有参考图且 API 支持 → 使用 Gemini 格式 (原生支持多模态)
  if (referenceImage && isGeminiApi(config.baseUrl)) {
    return generateImageGeminiWithReference(prompt, config, referenceImage);
  }

  // 有参考图但 API 不支持 → 仅用文本，忽略参考图
  if (referenceImage) {
    console.warn("Reference image provided but API does not support image input. Using text-only.");
  }

  return generateImage(prompt, config);
}

async function generateImageGeminiWithReference(
  prompt: string,
  config: ImageApiConfig,
  ref: NonNullable<GenerateImageOptions["referenceImage"]>,
): Promise<GenerateImageResult> {
  const url = `${config.baseUrl}/v1beta/models/${config.model}:generateContent`;
  const response = await fetch(url, {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "x-goog-api-key": config.apiKey,
    },
    body: JSON.stringify({
      contents: [{
        role: "user",
        parts: [
          { text: `Generate an image in this art style: ${prompt}` },
          {
            inlineData: {
              mimeType: ref.mimeType,
              data: ref.b64Data,
            },
          },
        ],
      }],
      generationConfig: {
        responseModalities: ["IMAGE", "TEXT"],
      },
    }),
    signal: AbortSignal.timeout(config.timeoutMs),
  });

  if (!response.ok) throw new Error(`IMAGE_HTTP_${response.status}`);
  const payload = await response.json();
  const parts = payload.candidates?.[0]?.content?.parts ?? [];
  for (const part of parts) {
    if (part.inlineData?.data) return { b64Data: part.inlineData.data };
  }
  throw new Error("IMAGE_NO_DATA_IN_RESPONSE");
}
```

```TypeScript
// lib/reference-image-manager.ts

import fs from "node:fs";
import path from "node:path";

interface CachedReferenceImage {
  b64Data: string;
  mimeType: string;
  hash: string;
}

const CACHE_DIR = path.resolve(process.cwd(), "data/cache/reference-images");

/**
 * 加载参考图并缓存 base64 数据。
 * 参考图路径相对于项目根目录。
 */
export function loadReferenceImage(imagePath: string): CachedReferenceImage {
  const absPath = path.resolve(process.cwd(), imagePath);
  const ext = path.extname(absPath).toLowerCase();
  const mimeType = ext === ".jpg" || ext === ".jpeg"
    ? "image/jpeg"
    : ext === ".webp"
      ? "image/webp"
      : "image/png";

  const buffer = fs.readFileSync(absPath);
  const b64Data = buffer.toString("base64");

  return { b64Data, mimeType, hash: "" };
}

    const absPath = path.resolve(process.cwd(), candidate);
```

## 8\. 自我辩证与反思



### 8\.1 风格锚定句的局限性



**问题**: 一个固定的语言描述能否真正"锚定"画风？



**反思**:

- 文本 prompt 对画风的控制力有限。同一段描述词在不同模型、不同 seed 下产出差异巨大。

- 风格锚定句能做到的是 **降低方差**，而非 **消除方差**。它把画风从"完全随机"拉到"大致一致"。

- 真正的画风一致性需要 **参考图输入** \(IP\-Adapter / style reference\) 或 **微调 LoRA**。

    

**应对**:

- 短期 \(本方案\): 用风格锚定句 \+ 预算系统，改善明显

- 中期: 探索 StepFun/Gemini 的参考图 API

- 长期: 如果画风一致性要求极高，考虑对生图模型做 LoRA 微调

    

### 8\.2 512 字符限制 vs 叙事丰富度



**问题**: 512 字符既要表达风格又要表达剧情，空间极其紧张。



**反思**:

- StepFun 的 512 字符限制是一个硬约束。当前 prompt 经常超限被截断。

- 风格锚定句 \~200 字符 \+ 负面提示 \~100 字符 = 300 字符已消耗，仅剩 \~210 字符给剧情。

- 这意味着剧情描述必须极其精炼，每章的核心视觉元素只能用 1\-2 句话表达。

    

**应对**:

- `buildBudgetedPrompt()` 确保高优先级内容不被截断

- 剧情描述应该聚焦 **视觉可渲染的元素**（场景、动作、光线），而非抽象叙事

- 负面提示可以压缩为固定短句，不需要每章重复

    

### 8\.3 参考图 vs Prompt 的权衡



**问题**: 当 API 同时接受参考图和文本 prompt 时，两者如何配合？



**反思**:

- 参考图传达的是"画风"，prompt 传达的是"内容"。理论上可以解耦。

- 但实际中参考图会影响构图、色调、甚至主体形态。如果参考图是哥特暗系角色，生成都市明亮场景可能会"串味"。

- 需要一个 `strength` 参数控制参考图的影响程度。

    

**应对**:

- 风格参考图选择 **背景/场景图** \(6fc098e3\.png\) 而非 **角色立绘** \(role1\.png\)，减少对主体的干扰

- `strength` 默认 0\.5\-0\.7，保留 prompt 对内容的主导权

- 在不同章节动态调整 strength: 开场高 \(建立风格\)，中段低 \(让剧情主导\)，结尾中 \(收束风格\)

    

### 8\.4 多风格 vs 统一风格



**问题**: 无限流副本可能跨越多种世界观，是统一画风还是动态切换？



**反思**:

- 完全统一会导致"所有副本看起来一样"，丧失世界观差异感。

- 完全动态会导致"同一副本内画风不一致"，破坏沉浸感。

- 最佳平衡: **风格框架统一，色调/光影动态调整**。

    

**应对**:

- "风格锚定句"定义不可变的框架 \(render\_style, line\_work\)

- "色调/光影/特效"根据主题动态变化

- 同一副本内保持一致，不同副本间有差异

    

### 8\.5 生产环境风险评估



|风险|概率|影响|缓解措施|
|---|---|---|---|
|风格锚定句在某些模型上无效|中|中|A/B 测试 \+ 备用锚定句|
|512 截断导致 prompt 语义断裂|高|中|预算系统 \+ 优先级排序|
|参考图 API 不可用|高|低|降级为纯文本方案|
|生成速度因参考图而变慢|中|低|异步生成 \+ 超时兜底|
|画风在某些剧情下不搭|中|中|主题→风格映射 \+ 人工调优|

## 附录 B: Prompt 模板对比示例



### 场景: 无限流副本第 3 章 — 雾中抉择



**当前基线 prompt** \(\~520 chars, 会被截断\):

```Plain Text
parallel life cinematic scene, absolutely no text or letters anywhere, stylized player avatar
inspired by uploaded visual style, avatar reference: fictional player silhouette, close shot,
direct gaze, intense decision moment, dramatic focus on expression, scene: 雾中抉择,
story context: 你站在浓雾弥漫的十字路口，四条道路通向不同的命运。远处传来低沉的钟声，
每一声都像是时间在倒数。你必须在雾气散去之前做出选择。visual echo from previous choice:
命运之钟, atmosphere: mysterious, cyan_blue, color saturation level: 0.7,
dramatic chiaroscuro, strong light-dark contrast, high stakes moment,
color influence: cool steel blue pressure undertone, style: cinematic illustration,
contemporary Chinese urban life, no text, no letters, no words, no Chinese characters,
no Japanese characters, no Korean characters, no typography, no subtitles, no captions,
no labels, no logo, no watermark, no public figure, no brand character, no identity document
```



**EFAC 风格锚定 prompt** \(\~498 chars\):

```Plain Text
eastern fantasy anime cinematic illustration, semi-realistic with painterly textures,
volumetric atmospheric lighting, cold-warm color contrast with dark base and luminescent accents,
ethereal narrative mood, high detail quality, dramatic chiaroscuro with selective warm glow,
close shot, direct gaze, intense decision moment, at a fog-shrouded crossroads,
four paths diverging into different fates, distant bell tolling, timer counting down,
thick mist and god rays, cold steel blue pressure undertone, mysterious atmosphere,
no text no letters no watermark no logo
```



**差异分析**:

- 基线: 空间被大量通用描述和冗长负面提示占据，剧情描述是中文（对英文生图模型无效）

- EFAC: 风格锚定优先，剧情核心聚焦可渲染的视觉元素，负面提示压缩为短句
