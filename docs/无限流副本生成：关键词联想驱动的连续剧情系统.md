## 一、问题定义：什么是"无限流副本"



### 1\.1 当前系统的限制



当前冒险系统是**一次性 5 章结构**：

```Plain Text
用户答题 → PersonaResult → 生成 AdventureDefinition (固定5章) → 用户选择 → 结束
```



**核心痛点：**

- 5 章结束后无法继续，用户需要"重新开始"

- 每次冒险是独立的，不引用之前的冒险经历

- 剧情不随用户的记忆和画像进化

- 无法形成"人生连续剧"的沉浸感

    

### 1\.2 无限流的目标



```Plain Text
第1次冒险 (5章) → 关键词提取 → 海马体记忆存储
                                    ↓
第2次冒险 (5章) ← 关键词联想 ← 记忆 + 画像 + 镜子
                                    ↓
第3次冒险 (5章) ← 关键词联想 ← 更丰富的记忆网络
                                    ↓
                                  ... 无限循环
```



**核心理念**：每次冒险结束后，系统从本次冒险中提取关键词（主题、选择倾向、情绪模式、未解决冲突），存入海马体记忆。下次冒险开始时，系统通过关键词联想从记忆网络中召回相关主题，生成与用户人生轨迹连贯的新副本。



### 1\.3 自我辩证：为什么不是真正的"无限"



**反对意见**：真正的无限流需要无限的叙事可能性，但 LLM 的上下文窗口有限，用户的人生主题也会收敛。



**回应**：

- "无限"不是指叙事无限，而是**系统不设硬性上限**

- 用户的人生主题会收敛，但**关键词组合**不会——同样的"职业焦虑"关键词，搭配不同的"记忆深度"（第1次 vs 第10次），会生成不同深度的副本

- 核心循环是**记忆积累 → 关键词丰富 → 剧情深化**，而非简单重复

## 二、核心机制：关键词联想引擎



### 2\.1 因果推理视角：关键词的本质



应用 core\-dimension\-distribution\-causal\-reasoning 的分析框架：



**Step 1 \- 业务问题定义：** 如何从用户的历史数据中提取能驱动新剧情的关键词？



**Step 2 \- 锁定粒度：** 一个关键词 = 一个可复用的叙事单元。粒度太粗（"职业"）无法驱动具体剧情，粒度太细（"周三下午的焦虑"）无法复用。



**Step 3 \- 维度剖面：**



|关键词来源|粒度|示例|复用频率|
|---|---|---|---|
|21维画像 \(O1,C1,E1\.\.\.\)|粗|"高开放性"|每次都用|
|潜意识镜子 \(星座/五行\)|中|"火象能量"|每次都用|
|海马体记忆 \(Concept\)|中|"与上司的冲突"|高频复用|
|冒险选择 \(branch\_vector\)|细|"倾向稳定"|持续追踪|
|未解决冲突 \(threads\)|细|"未完成的项目"|直到解决|



**Step 4 \- 分布诊断：**

- 关键词应该**幂律分布**：少数核心主题（职业焦虑、关系冲突）高频出现，多数具体事件低频

- 如果关键词过于集中（Top1 \> 80%），说明用户陷入了**重复模式**，系统应主动引入新维度

- 如果关键词过于分散（Top10 \< 30%），说明用户在**回避核心问题**，系统应聚焦

    

**Step 5 \- 映射基数：**

- `关键词 → 副本主题` 应该是 1:N（一个关键词可以驱动多个不同副本）

- `副本 → 关键词` 应该是 N:1（一个副本产生多个关键词）

- 这是**多对多**关系，需要一个中间层（关键词图谱）

    

**Step 6 \- 跨层对比：**

- 源层（用户输入）：有丰富的人生信息

- 结果层（当前冒险）：只用了 Quiz \+ MysticProfile，未用海马体记忆

- **差距**：海马体记忆中积累的用户模式未被冒险系统消费

    

**Step 7 \- 因果结论：**

- **Action Bucket**: \#6 Missing bridge layer — 需要一个"关键词联想层"桥接海马体记忆和冒险生成

```TypeScript
interface KeywordNode {
  id: string;                    // 唯一标识
  keyword: string;               // "职业焦虑" / "火象能量" / "回避冲突"
  source: KeywordSource;         // 来源类型
  dimension: string;             // 关联的画像维度 (如 "O1", "D4")
  strength: number;              // 强度 0-1 (衰减)
  first_seen: number;            // 首次出现时间
  last_seen: number;             // 最后出现时间
  occurrence_count: number;      // 出现次数
  metadata: Record<string, any>; // 来源特定数据
}

interface KeywordEdge {
  from_keyword: string;
  to_keyword: string;
  relation: "co_occurs" | "causes" | "contradicts" | "evolves_to";
  strength: number;              // 共现/因果强度
  context: string;               // 关系上下文
}

type KeywordSource =
  | "taxonomy"       // 21维画像
  | "mirror"         // 潜意识镜子
  | "memory"         // 海马体记忆 (Concept)
  | "adventure"      // 冒险选择/结果
  | "thread"         // 未解决冲突
  | "emotion"        // 情绪模式
  | "milestone";     // 人生里程碑
```

### 3\.2 关键词联想算法



```TypeScript
// lib/life-game/keyword-association.ts

interface AssociationResult {
  keyword: string;
  source: KeywordSource;
  score: number;          // 综合得分
  reason: string;         // 为什么选中这个关键词
  freshness: number;      // 时间新鲜度 0-1
  relevance: number;      // 与当前情境的相关性 0-1
  novelty: number;        // 新颖度 0-1 (与上次副本的差异)
}

async function associateKeywords(
  profile: ProfileContext,
  memory: MemoryGraph,
  lastAdventure: AdventurePathMemory | null,
  destiny: DestinySnapshot,
): Promise<AssociationResult[]> {
  const candidates: AssociationResult[] = [];

  // 1. 画像关键词 (每次必选, 高权重)
  const profileKeywords = extractProfileKeywords(profile);
  candidates.push(...profileKeywords.map(k => ({
    ...k,
    score: 0.9,
    reason: "核心人格画像",
    freshness: 1.0,
    relevance: 0.9,
    novelty: 0.1,  // 画像不变，新颖度低
  })));

  // 2. 海马体记忆关键词 (按 strength 衰减)
  const memoryKeywords = extractMemoryKeywords(memory);
  candidates.push(...memoryKeywords.map(k => ({
    ...k,
    score: k.strength * 0.7,
    reason: `记忆网络中的高频主题 (出现${k.count}次)`,
    freshness: timeDecay(k.last_seen, 30),  // 30天半衰期
    relevance: k.strength,
    novelty: 0.5,
  })));

  // 3. 上次冒险的行为倾向 (如果有)
  if (lastAdventure) {
    const behaviorKeywords = extractBehaviorKeywords(lastAdventure);
    candidates.push(...behaviorKeywords.map(k => ({
      ...k,
      score: 0.6,
      reason: "上次冒险的选择倾向",
      freshness: 1.0,
      relevance: 0.7,
      novelty: 0.3,
    })));
  }

  // 4. 未解决冲突 (高优先级, 直到解决)
  const threadKeywords = extractThreadKeywords(lastAdventure);
  candidates.push(...threadKeywords.map(k => ({
    ...k,
    score: 0.8,
    reason: "未解决的人生悬念",
    freshness: 1.0,
    relevance: 0.8,
    novelty: 0.2,  // 悬念不新颖但必须处理
  })));

  // 5. 因果过滤 (core-dimension-distribution-causal-reasoning)
  const filtered = causalFilter(candidates, {
    // 避免重复: 如果上次副本主题是"职业焦虑"，这次降低其权重
    deduplicateAgainst: lastAdventure?.active_threads || [],
    // 聚焦: 如果某类关键词占比过高 (>60%)，引入新维度
    diversifyThreshold: 0.6,
    // 排除已解决: 不再召回已 resolved 的冲突
    excludeResolved: lastAdventure?.resolved_threads || [],
  });

  // 6. 排序: score * freshness * relevance * (1 + novelty*0.3)
  return filtered
    .sort((a, b) => {
      const sa = a.score * a.freshness * a.relevance * (1 + a.novelty * 0.3);
      const sb = b.score * b.freshness * b.relevance * (1 + b.novelty * 0.3);
      return sb - sa;
    })
    .slice(0, 8);  // Top-8 关键词
}
```



### 3\.3 因果过滤器



基于 core\-dimension\-distribution\-causal\-reasoning 的因果推理模板：



```TypeScript
function causalFilter(
  candidates: AssociationResult[],
  context: {
    deduplicateAgainst: string[];
    diversifyThreshold: number;
    excludeResolved: string[];
  }
): AssociationResult[] {
  // Template A: 大量 "无" 桶 → 检查是否遗漏了关键数据源
  const sourceCoverage = new Map<KeywordSource, number>();
  for (const c of candidates) {
    sourceCoverage.set(c.source, (sourceCoverage.get(c.source) || 0) + 1);
  }
  // 如果某个来源贡献了 >80% 的关键词，标记为偏差
  for (const [source, count] of sourceCoverage) {
    if (count / candidates.length > 0.8) {
      // 注入多样性: 从其他来源补充
    }
  }

  // Template C: 强一对多映射 → 检查是否需要主题聚焦
  // 如果 Top-3 关键词占总 score >70%，说明主题过于集中
  const sorted = [...candidates].sort((a, b) => b.score - a.score);
  const top3Score = sorted.slice(0, 3).reduce((s, c) => s + c.score, 0);
  const totalScore = candidates.reduce((s, c) => s + c.score, 0);
  if (top3Score / totalScore > context.diversifyThreshold) {
    // 从低频关键词中补充多样性
  }

  // Template D: 竞争基表 → 排除已解决主题
  return candidates.filter(c =>
    !context.excludeResolved.includes(c.keyword)
  );
}
```

## 五、关键词图谱存储方案



### 5\.1 存储选型



|选项|优势|劣势|
|---|---|---|
|SQLite JSON 列|简单，与现有系统一致|无法向量检索|
|LanceDB 向量表|支持语义检索|需要额外依赖|
|**SQLite \+ LanceDB 混合**|**结构化查询 \+ 语义检索**|**需要同步**|



**选择混合方案**：SQLite 存储关键词元数据（节点、边、强度），LanceDB 存储关键词向量（支持语义扩展）。



### 5\.2 SQLite 表结构



```SQL
-- 关键词节点
CREATE TABLE keyword_nodes (
  id TEXT PRIMARY KEY,
  keyword TEXT NOT NULL,
  source TEXT NOT NULL,          -- taxonomy/mirror/memory/adventure/thread
  dimension TEXT,                -- 关联的画像维度
  strength REAL DEFAULT 1.0,
  first_seen REAL NOT NULL,
  last_seen REAL NOT NULL,
  occurrence_count INTEGER DEFAULT 1,
  metadata TEXT DEFAULT '{}',    -- JSON
  group_id TEXT DEFAULT ''
);

-- 关键词边（共现/因果关系）
CREATE TABLE keyword_edges (
  from_keyword TEXT NOT NULL,
  to_keyword TEXT NOT NULL,
  relation TEXT NOT NULL,        -- co_occurs/causes/contradicts/evolves_to
  strength REAL DEFAULT 1.0,
  context TEXT DEFAULT '',
  group_id TEXT DEFAULT '',
  PRIMARY KEY (from_keyword, to_keyword, group_id)
);

CREATE INDEX idx_keyword_source ON keyword_nodes(source, group_id);
CREATE INDEX idx_keyword_strength ON keyword_nodes(strength DESC);
CREATE INDEX idx_edge_relation ON keyword_edges(relation);
```



### 5\.3 LanceDB 向量表



```TypeScript
const KEYWORD_VECTORS_SCHEMA = pa.schema([
  pa.field("keyword_id", pa.utf8()),
  pa.field("keyword", pa.utf8()),
  pa.field("keyword_vector", pa.list_(pa.float32(), list_size=512)), // CLIP text encoding
```

## 八、关键设计决策



### 决策 1: 关键词粒度



**选择**：中等粒度（"职业焦虑"而非"工作"或"周三下午的焦虑"）

**理由**：太粗无法驱动具体剧情，太细无法复用。中等粒度允许同一关键词在不同上下文中产生不同副本。



### 决策 2: 关键词衰减策略



**选择**：指数衰减，半衰期 30 天

**理由**：与海马体记忆的遗忘曲线一致。30 天前的关键词权重降至 50%，90 天前降至 12\.5%。但"未解决冲突"关键词不衰减，直到解决。



### 决策 3: 多样性 vs 聚焦



**选择**：Top\-3 关键词占比上限 60%，超过时从低频关键词中补充

**理由**：如果用户反复遇到"职业焦虑"，系统应主动引入新维度（如"关系"、"健康"），避免剧情固化。这对应因果推理中的 Template C（强一对多映射）。



### 决策 4: LLM 调用次数



**选择**：每次副本 2 次 LLM 调用（主题生成 1 次 \+ 路径预演 1 次）

**理由**：路径预演合并为一次调用（所有路径一起生成），而非每个路径单独调用。减少延迟和成本。



### 决策 5: 记忆注入时机



**选择**：在路径预演 Prompt 中注入，而非主题生成

**理由**：主题生成需要聚焦于"用户当前的关键词"，而路径预演需要"参考用户的历史模式"。记忆注入过早会干扰主题的原创性。

## 九、自我辩证与风险



|风险|概率|影响|缓解|
|---|---|---|---|
|关键词固化（用户反复看到相同主题）|中|高|多样性过滤 \+ 主动引入新维度|
|记忆注入导致剧情重复|中|中|"上次冒险回顾"只展示差异点|
|LLM 生成质量不稳定|中|高|结构化 JSON 输出 \+ fallback 模板|
|关键词图谱膨胀|低|低|定期清理低强度节点|
|用户感到"被看穿"|低|中|记忆注入措辞温和，不直接引用原话|
|无限流循环疲劳|中|中|每 5 次副本后生成"人生阶段总结"|
